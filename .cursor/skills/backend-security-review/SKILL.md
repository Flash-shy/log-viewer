# 后端安全规范审查（只读）

## 适用范围

- **纳入范围**：后端代码与紧邻配置中的**安全规范符合性**与**常见漏洞模式**（见下表各维度）；以**静态阅读**为主，可结合项目内测试脚本结果（若用户已提供）。
- **排除范围**：不替代专业渗透测试或 SCA 工具的完整报告；**不默认修改代码**。若用户要求「仅审查」，须**禁止**提交补丁或重构；修复可另起对话并引用编号。
- **敏感信息**：报告中**禁止**回显完整密钥、Token、密码；须掩码（如 `sk-***…last4`）。

## 工作流程

1. **明确范围**：若用户未指定路径或模块，询问待审查的目录、文件或服务边界。
2. **只读分析**：使用读取、搜索、语义搜索；不得擅自改代码，除非用户明确要求修复且与本 Skill 的「仅审查」指令不冲突。
3. **按维度归纳**：使用下文「类型代码与检查要点」；若用户限定某几类，**不得**在同一轮扩展到未要求的类型。
4. **统一编号**：每发现一项使用 **类型代码-序号**（如 `AUTHZ-1`），每一类型内序号从 1 起连续递增。
5. **空结果**：某维度检视后无问题时，在该类型下写一条子项 **「未发现明显问题（基于当前静态检视）」**，或明确说明**未覆盖**的运行时行为。

## 类型代码与检查要点

| 代码 | 维度 | 最多条数 | 检查要点（示例） |
|------|------|----------|------------------|
| **AUTH** | 认证与会话 | 10 | 密码哈希算法与参数（如 bcrypt/scrypt/argon2）、会话固定、Cookie `Secure`/`HttpOnly`/`SameSite`、JWT `alg`/过期/撤销、API Key 传递方式、暴力破解防护、注销与会话失效 |
| **AUTHZ** | 授权与访问控制 | 10 | 资源级权限（防 IDOR）、角色/策略一致性、默认拒绝、批量接口越权、管理端与普通端隔离、文件/对象存储路径穿越 |
| **INJ** | 注入与命令执行 | 10 | SQL 拼接与预编译、NoSQL/LDAP/OS 命令注入、模板注入、反序列化不可信数据 |
| **SECRET** | 密钥与凭据 | 逐项 | 硬编码密钥/Token、默认口令、仓库内 `.env` 提交、调试接口暴露凭据；**位置 + 风险级（高/中/低）**，值须掩码 |
| **CRYPTO** | 加密与传输 | 8 | TLS 版本与证书校验、随机数（`crypto/rand` 等）、对称加密模式（避免 ECB）、敏感字段是否落盘加密 |
| **INPUT** | 输入与文件 | 10 | 类型/长度/范围校验、路径与文件名白名单、上传类型与大小限制、SSRF 可调 URL、开放重定向 |
| **ERR** | 错误与信息泄露 | 8 | 堆栈/内部路径返回给客户端、过于细分的错误枚举（便于枚举用户）、生产环境调试开关 |
| **LOG** | 日志与审计 | 8 | 密码/Token/PII 全量写日志、审计缺失（敏感操作无记录）、日志注入 |
| **DEP** | 依赖与供应链 | 6 | 已知 CVE、过时依赖、`replace`/`override` 风险、锁文件完整性（说明需结合 `govulncheck`/npm audit 等时标注**工具名**） |
| **HTTP** | HTTP 安全与跨域 | 8 | `CORS` 是否过宽、`CSRF` 对浏览器会话类请求、`Content-Security-Policy`（若适用）、`X-Content-Type-Options` 等安全头 |
| **API** | API 滥用防护 | 8 | 限流/熔断、幂等与重放、批量与导出接口滥用、GraphQL depth/introspection（若适用） |
| **OPS** | 部署与运维 | 6 | 生产关闭 debug、最小权限运行、环境变量注入密钥、健康检查与指标是否泄露内部信息 |

若用户要求**全维度扫描**，按上表逐类检视并遵守条数上限；可合并高度重复的同类问题为一条并注明多处位置。

## 严重级别（每条子项须标注）

在子项内使用其一：**严重** / **高** / **中** / **低** / **信息性**，并简述**可利用条件**（如「需已认证」「需内网」）。

## 输出格式（必选）

采用**两级列表**：

- **第一级**：`- **代码 · 简短标题**`（如 `- **AUTH · 认证与会话**`）
- **第二级**：每条以 **代码-序号** 开头，须含：
  - **位置**：文件路径，必要时函数/处理器名
  - **问题**：违反的规范或风险
  - **严重级别** + **简要缓解方向**（不写完整补丁，除非用户另行要求）

示例：

```markdown
## 后端安全规范审查（只读）

**审查范围**：`<路径>`

- **AUTH · 认证与会话**
  - **AUTH-1** … **严重** …
- **INJ · 注入与命令执行**
  - **INJ-1** … **高** …
```

## 与「代码坏味道」Skill 的分工

- **discover-code-smells** 侧重可维护性结构（魔法数、超长方法等）；其中的 **SEC** 仅覆盖敏感信息硬编码。
- 本 Skill 覆盖**完整后端安全维度**；若用户同时关心结构与安全，可分两轮或要求先安全后结构。

## 结语

本 Skill 产出**可追踪的编号清单**，便于在 Issue/PR 中引用；**修复与复测**应作为单独任务，并可配合自动化扫描与人工验证。
